# DPDP Scan Engine Integration Guide

## Purpose

This project is a backend-driven DPDP data scanning and reporting application. It scans a structured data source, profiles tables and columns, detects PII exposure and masking gaps, builds governance and RAID insights, and produces downloadable reports for downstream applications.

This guide is for the team integrating this application into another product.

## How the Backend Works

The main workflow is defined in [backend/main.py](<e:\SLP1\Swarna_P\DI_LABS\Automated_Data_Discovery_Platform\test\scan_DPDP\backend\main.py>).

High-level pipeline:

1. `data_agent`
   Builds `source_inventory`, `schema`, and `schema_analysis` from the connected database.
2. `profiling_agent`
   Samples data and computes profiling and DQ metrics.
3. `classification_agent`
   Infers column-level intelligence such as PII and masking status.
4. `document_insights_agent`
   Compares structured metadata to document-driven expectations.
5. `unstructured_agent`
   Scans unstructured files for PII findings.
6. `gap_analysis_agent`
   Builds findings and recommendations.
7. `risk_agent`
   Converts issues into risks.
8. `raid_agent`
   Produces RAID-style outputs.
9. `governance_agent`
   Adds governance and relationship insights.
10. `aggregator`
    Produces the final output and export artifacts.

## API Surface

The API entrypoint is [backend/api/app.py](<e:\SLP1\Swarna_P\DI_LABS\Automated_Data_Discovery_Platform\test\scan_DPDP\backend\api\app.py>).

Current endpoints:

- `GET /`
  Health-style response.
- `GET /scan`
  Starts an async scan job using the current backend workflow.
- `GET /status/{job_id}`
  Returns job status.
- `GET /result/{job_id}`
  Returns the completed result payload.
- `GET /scan/history`
  Returns monitoring history.
- `GET /monitoring/history`
  Returns monitoring history.
- `POST /scan/schedule`
  Enables or disables scheduled scans.
- `POST /tasks/create`
  Creates remediation tasks.
- `POST /tasks/update/{task_id}`
  Updates remediation tasks.
- `GET /tasks`
  Lists remediation tasks.
- `GET /exports/{job_id}/{format_name}`
  Downloads generated reports in `json`, `pdf`, `pptx`, `docx`, `xlsx`, `md`, or `txt`.

## Current Configuration Behavior

The structured scan currently depends on hardcoded database configuration in [backend/config.py](<e:\SLP1\Swarna_P\DI_LABS\Automated_Data_Discovery_Platform\test\scan_DPDP\backend\config.py>).

Current assumptions:

- The scanner connects directly to PostgreSQL.
- Connection values are read from `DB_CONFIG`.
- `GET /scan` starts the workflow without accepting an external input payload.
- `document_insights_agent` currently reads document context from local filesystem folders:
  - `storage/document_insights`
  - `storage/templates`
  - `storage/assembler`

## Important Integration Notes

### 1. Remove `test_data` from the integration flow

The `test_data/` folder is only sample SQL for local testing and should not be used in production integration.

The integrating team should:

- Remove any dependency on `test_data/`
- Avoid seeding scans from local SQL scripts
- Point the application to real upstream metadata and source systems

### 2. Connect the app to real Source Inventory inputs

The most important integration change is to replace the current source discovery behavior with Source Inventory input from the parent application.

Today:

- `backend/agents/data_agent.py` introspects the connected database and builds `source_inventory` internally.

For integration:

- The parent application should provide Source Inventory as the authoritative input.
- The scan flow should consume that input instead of assuming one local PostgreSQL database.

Expected `source_inventory` shape is effectively:

```json
{
  "source_type": "postgresql",
  "database": "source_db_name",
  "tables": [
    {
      "schema": "public",
      "table": "customers",
      "qualified_name": "public.customers",
      "owner": "team_or_user",
      "dataset_owner": "team_or_user",
      "data_steward": "owner_name_or_group",
      "ownership_status": "resolved",
      "columns": [
        {
          "name": "customer_id",
          "type": "INTEGER",
          "nullable": false,
          "default": null,
          "primary_key": true
        }
      ],
      "primary_key": ["customer_id"],
      "foreign_keys": [],
      "indexes": []
    }
  ]
}
```

Recommended integration approach:

- Add a new scan endpoint that accepts a request body containing Source Inventory and any related context.
- Update `data_agent` so it uses incoming `state["source_inventory"]` when supplied.
- Fall back to DB introspection only for local development if you still want that mode.

### 3. Connect DIAS output into document processing

The project already has a document alignment stage, but it currently reads JSON files from local storage.

Today:

- [backend/services/document_adapters.py](<e:\SLP1\Swarna_P\DI_LABS\Automated_Data_Discovery_Platform\test\scan_DPDP\backend\services\document_adapters.py>) loads document metadata from local directories.
- `document_insights_agent` calls `normalize_document_context(...)` using local files.

For integration:

- Replace local file loading with DIAS output from the parent application.
- DIAS should provide the document-derived rules, schema expectations, and relationship expectations.

The three DIAS-fed logical inputs are:

- `document_insights`
  Rule-level expectations such as masking, required fields, and expected types
- `expected_schema`
  Template-driven required fields and schema expectations
- `relationships`
  Expected source-to-source relationships and join expectations

Recommended integration approach:

- Pass DIAS output into the workflow state, for example as `state["document_context"]`
- Update `document_insights_agent` to prefer incoming `state["document_context"]`
- Keep file-based adapters only as a local fallback for testing

Target document context shape:

```json
{
  "document_insights": [
    {
      "doc_id": "policy_001",
      "document_type": "document_insight",
      "rules": [
        {
          "field": "email",
          "masking": "MASKED",
          "required": true,
          "data_type": "varchar"
        }
      ]
    }
  ],
  "expected_schema": [
    {
      "template_id": "customer_template",
      "expected_schema": [
        {
          "field": "email",
          "required": true,
          "data_type": "varchar"
        }
      ]
    }
  ],
  "relationships": [
    {
      "doc_id": "assembler_001",
      "from_table": "public.orders",
      "to_table": "public.customers",
      "field": "customer_id",
      "relationship_type": "expected"
    }
  ]
}
```

### 4. API change recommended for real integration

`GET /scan` is fine for demo use, but it is not the right interface for a real integration because there is no request payload.

Recommended replacement:

- Add a `POST /scan` endpoint
- Accept:
  - Source Inventory input
  - Optional profiling or scan options
  - DIAS output or document context
  - Optional unstructured document references

That will let the integrating application control exactly what is scanned and compared.

## Known Integration Constraints

- Database credentials are currently hardcoded in `backend/config.py`
- The workflow currently assumes direct DB connectivity
- Document inputs are currently file-based, not API-driven
- The API starts scans asynchronously but does not yet accept external scan payloads
- Runtime artifacts are written to local filesystem storage

## Recommended Pre-Production Cleanup

Before integrating into another application, the team should do these changes:

1. Remove `test_data/` from the runtime path and deployment artifact.
2. Externalize DB config into environment variables or secret management.
3. Add a `POST /scan` request contract for Source Inventory and DIAS payloads.
4. Make `document_insights_agent` consume DIAS output directly.
5. Keep filesystem adapters only as a local development fallback.
6. Review `storage/` write paths if the target environment uses object storage or shared storage instead of local disk.
